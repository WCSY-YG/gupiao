## 自动化能力说明

本仓库按“可恢复推进”方式运行：每一轮启动时读取项目计划、项目记忆和任务清单，选择下一项可执行任务，完成后更新状态并提交本地 Git。GitHub 推送是同步步骤；若凭据或权限不可用，记录为 `push_pending`，但不阻塞下一项任务。

当前 Codex 环境未暴露 App 级 `automation_update` 定时任务工具，因此本文件先作为项目内自动化协议。后续一旦定时工具可用，应按本文件配置后台唤醒；在此之前，任何线程恢复或新一轮执行都必须遵循同样规则。

## 每轮执行流程

1. 读取 `docs/PROJECT_MEMORY.md`、`docs/PROJECT_PLAN.md`、`docs/PROJECT_TASKS.md`。
2. 检查 `git status --short`，确认是否存在用户未提交改动。
3. 从 `docs/PROJECT_TASKS.md` 选择第一个 `pending` 且依赖已完成的任务。
4. 开始任务前，将该任务状态改为 `in_progress` 并记录时间。
5. 执行任务，优先保持小步、可验证、可提交。
6. 完成后运行相关检查；若无法运行，必须记录原因。
7. 将任务状态改为 `done`，更新完成记录和 `docs/PROJECT_MEMORY.md` 的下一步。
8. 执行 Git 更新：
   - `git status --short`
   - `git add` 本轮变更文件
   - `git commit -m "<type>: <summary>"`
   - 若 GitHub 凭据可用，再执行 `git push origin <current-branch>`
   - 若推送失败或已知凭据不可用，在 `docs/PROJECT_MEMORY.md` 记录 `push_pending`，继续下一项任务
9. 若线程因完成本轮任务而停止，下一轮从第 1 步继续，直到所有任务为 `done`。

## 重要变更范围

以下变更必须记录到任务清单或项目记忆：

- 任务状态变化：`pending`、`in_progress`、`done`、`blocked`。
- 项目规划、里程碑、技术栈或风险边界变化。
- 数据源、依赖、存储模型、回测假设变化。
- 新增 CLI/API/UI 行为。
- 测试、构建、回测或数据质量检查失败。
- GitHub 推送失败、`push_pending` 状态或需要人工凭据/权限处理。

## 阻塞处理

如果任务被阻塞：

1. 将任务状态改为 `blocked`。
2. 在 `docs/PROJECT_TASKS.md` 记录阻塞原因和所需外部输入。
3. 如果存在不依赖该阻塞项的后续任务，可以继续推进。
4. 如果没有可继续任务，停止并向用户说明具体阻塞点。

## 默认检查节奏

- 线程启动或恢复时：立即检查并推进下一项任务。
- 每完成一项任务后：立即更新任务清单并提交；GitHub 推送可用时再同步，失败不阻塞后续任务。
- App 级定时工具可用时：建议设置为每天 09:00 CST 检查一次；若上一轮仍未完成，则继续当前 `in_progress` 任务。

## 安全边界

- 本项目仅用于研究和辅助分析，不输出投资承诺，不默认接入真实交易。
- 回测与信号必须持续记录数据质量、未来函数、幸存者偏差、财务数据偷看和过拟合风险。
- 不覆盖用户未提交改动；如有冲突，必须先说明并协同处理。
